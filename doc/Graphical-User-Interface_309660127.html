DTK : Graphical User Interface

Created by Sernandez Martinez, Felix on Oct 02, 2019

The Deployment Tool Kit (DTK) is divided in frames. It contains 4 main frames.

1 - Main Frame

The application has a main frame, this will remain opened, and lets the user reach the different functionalities:


Org Management:

  • Organizations: It will open the Organization Frame, where the user can manage the different salesforce environments and sandboxes.


Release Management:

  • Retrieve: It will open a Retrieve Frame, where metadata retrieves can be performed from the already configured environments.
  • Deploy: It will open a Deploy Frame, where metadata deploys can be performed from Zip/Folder/Git into the already configured environments.


This Main windows tool bar gives access to the Edit Menu, containing Settings and some customization.


1.1 - Settings

The Settings menu contains parameters and default values.


Use Advanced DTK: Check this to display all advanced controls.

Unlock Source:Â Check this to display Source textbox in Retrieve/Deploy frames. In the case of Retrieve, source field will allow to select the target of the retrieve, source value by default is Config. Source field is used in DTK Scripts, it should contain a Sandbox name of the selected Organization. Instrucctions with SOURCE will target that environment.

Unzip Retrieved: Check this to check by default the Unzip Retrieved checkbox on Retrieve frame.

Prefix(ex) to Exclude Package(s): Prefix of the package to exclude when retrieving metadata. Separate it by comma if more than one needs to be set.

Default Metadata Folder: Default metadata folder to be included when deploying from git.

Default Pre-deploy Data Script: Default pre-deploy data script to be included when deploying from git.

Default Pro-deploy Data Script: Default post-deploy data script to be included when deploying from git.

Default API Version: Default API version to be used in the retrieve and deploy. Execute 'sfdx force' in a terminal to see your latest version installed.


1.2 - Metadata Templates & SOQL Templates

Both Templates frames has the same interface, this being divided in two tabs: Manage Templates and Add Templates.

Metadata Templates:



Metadata Templates are used in the Retrieve and Deploy modules. This metadata templates makes easier to deploy different sets of metadata depending on the character of the operation the user is making.

The DTK also comes with 4 default Metadata Templates: All, None, Profiles and Related, Typical.

"All" and "None" metadata templates can't be edited or deleted.


After Adding or updating a Metadata Template it will be available in Retrieve and Deploy dropdowns fields for them use.

Pro Tip: This information is saved in JSON format in metadataTemplates.json file in .dtkconfig folder, located in your %User% documents folder in your PC.


SOQL Templates:

SOQL Templates can be used in the Retrieve module in the Export Tab. This templates lets the user save some useful queries in the DTK that he may use frequently to extract some data.

The DTK also comes with one default SOQL Template (Custom Settings Migration). This SOQL query extracts the useful information in the Custom Setting Migration in a format that is useful for merging Custom Settings with new versions.

In the example picture I add "Territories" SOQL Template, it extract the ID, Name, and DeveloperName from the table Territory.

Pro Tip: This information is saved in JSON format in SOQLTemplates.json file in .dtkconfig folder, located in your %User% documents folder in your PC.

2- Organizations Frame


Clicking on the Organization Management â Organizations Button in the Main Frame we find the Organizations Frame:

This frame has 4 tabs

Add Organization:Â This tab lets the user add a new Salesforce organization into the DTK.

Organization: Name of Organization entry.

GIT URL: This field is not required, this can be used to link a git based version control repository with the org.

GIT User: This field is not required. Self-explanatory.

GIT Password: This field is not required. In case the git is located in GITLAB and user has enabled Two-Factor Authentication, user will need to create a API Token in his Gitlab Profile and use the Token instead of the password. (Personal access tokens)

Metadata Folder: filled by default with the standard folder structure for Salesforce projects that use Metadata API.

Pre-deploy Script: Default location for the Pre-deploy script. This script is in DTK scripting language and will be executed before the deploy is applied in a Deploy procedure.

Post-deploy Script: Default location for the Post-deploy script. This script is in DTK scripting language and will be executed after the deploy is applied in a Deploy procedure.


Manage Organizations: This tab lets the user edit the added organizations settings.

This frame presents same fields as in Add Organization but with the purpose of letting the user Update or Delete an Organization.

Update button: This button will save the settings that are showing on screen for the selected Organization.

Delete button: This button will delete the selected Organization.


Add Sandbox:

This tab lets the user create Sandboxes in the DTK app.

Organization: Selected Organization will be the one containing the Sandbox after creation.

Sandbox Type: This will be the sandbox entry name in the DTK. It offers 4 values by default: Config, QA, UAT and Prod.

Only one Production Sandbox can be added for each Organization. The name of this sandbox will always be Prod.


*SandBox Override Name ( Advanced DTK): This text field is just like Sandbox Type, and if you use it to write a name, it will override the Sandbox Type value. Prod Orgs can't be created while this field filled.

All sandboxes created by Sandbox Override name will point to test.salesforce.com.

Add Sandbox: this button will use SFDX CLI that authorizes a Salesforce org by opening a browser so you can log in through salesforce.com.

You can follow instructions on the Log. This button will open a browser window ( with your default browser). If "Prod" is the name selected the URL opened will be login.salesforce.com. For other sandbox types the URL will be test.salesforce.com.

When web loads, you can log into the desired environment. If the credentials are correct, then Salesforce Environment will ask permission for CLI:



Just click Allow.

When this process is finished. Log in Organizations frame will show a confirmation and will let you close that browser window.

From now on this sandbox will be connected with DTK app in this PC. This connection will remain usable even when the password of the admin user used is changed.






Show SFDX Alias List button: this button will show all Auth conexions user has in the machine.



Manage Sandboxes:Â 


This tab Lists the saved Sandboxes for each Organization and lets the user Open then in browser already loged in or delete the conexion and entry from DTK.

Organization: Lists the available organizations.

Sandboxes: Lists the available sandboxes for the organization selected.

Open button: Will open and log in the sandbox selected in a browser window.

Delete Sandbox button: Will delete the sandbox entry in DTK.


3- Retrieve Frame

This frame will let the user extract Metadata and Data from Salesforce Environments. It will use the API version selected in the Settings.

First screenshot shows the frame by default (with Metadata tab selected). Second screenshot shows the frame with Unlock Source setting selected (with Export Data tab selected). Please note that Source field is in the header grey section and will let the user Select a Source Environment between the Sandboxes. The Source value by default is Config.


Organization: lets the user select the target Organization.

*Source (Unlock Source setting): lets the user select the Sandbox.

Metadata tab: contain all the functionalities and parameters needed to retrieve metadata from a Salesforce environment. It lets the user select what metadata types wants to extract with Metadata Tempale and Metadata Types selection boxes, but also lets the user select it's own manifest file for the retrieve.

  • Retrieve Type: Metadata Retrieve or Manifest File. On this selection depends the following fields.
    • Metadata Retrieve
      • Metadata Template: list the templates available, this templates are created, edited and deleted in Settings â Metadata Templates.
      • Metadata Types: will show the configuration of the metadata template selected. User can edit this templates "on the go" for the retrieve, but this will not modify the metadata template.
    • Manifest File:
      • Manifest File - Browse: This button will let the user select a package.xml file ( Manifest File) to be used on the retrieve. This file will contain an API version that will override the DTK setting.

  • Exclude Packages: this text field lets the user add words separated with "," in order to exclude those metadata from the retrieve. By default et4ae5 is excluded (Marketing metadata)
  • Unzip Retrieved: this will be checked by default when setting is checked. By default SFDX retrieve will download a zip with the metadata. With this option checked, this zip will be unziped when the download finishes.
  • Clear Logs: while this checkbox is checked the Log window will be cleared when Retrieve button is clicked.
  • Wait time: this box acceps number input. It will let the user select a waiting time in minutes for the retrieve Job. By default it contains "-1" this mean infinite waiting.

This Retrieve frame will wait until retrieve is finished, refreshing automatically the log, and unziping the metadata at the end. If other number is selected, then DTK will only wait the number provided in minutes, while it is waiting it will refresh automatically the log and unzip the metadata. If the wait time expires then the refresh log will be unlocked, allowing the user to change the environment active in the header and retrieving from another org ( we recommend to do this in another Retrieve frame window separately).


  • Retrieve button: this button will launch the Retrieve. Files will be downloaded on the Workspace selected ( it is selected automatically on Organization - Source selection) and can be seen on the Log.
  • Log: will show relevant information regarding the retrieve process.
  • Stop button: will stop the retrieve process as soon as possible for SFDX.
  • Refresh Log button: this will launch a SFDX report command in order to receive some feedback about the current retrieve status. This option is not available when waiting the retrieve, as it is made automatically.

Export Data tab: contain all the functionalities and parameters needed to extract data from a Salesforce environment, using SOQL queries.

  • File Name: name of the file that will contain the results of the query. It will be saved on the Workspace selected ( it is selected automatically on Organization - Source selection) and can be seen on the Log. This file will be ".csv" and will have comma separator ',' and use double quotes '"' only when needed.
  • SOQL Templates: list the templates available, this templates are created, edited and deleted in Settings â SOQL Templates.
  • SOQL Query: lets the user edit and review the query before execution.
  • Retrieve button: executes the query and saves the info in the File selected in File Name. Process can be followed in the log.
  • Log: will show relevant information regarding the extract process.
  • Stop button: will stop the extract process as soon as possible for SFDX.
  • Refresh Log button: this will launch a SFDX report command in order to receive some feedback about the current retrieve status.



4- Deploy Frame


This frame will let the user Deploy Metadata and execute DTK Scripts in Salesforce Environments. It will use the API version selected in the Settings.

First screenshot shows the frame by default. Second screenshot shows the frame with Unlock Source setting and Use Advanced DTK setting selected. Please note that Source field is in the header grey section and will let the user Select a Source Environment between the Sandboxes. The Source value by default is Config.

In this case, the advanced DTK also lets the user choose the Deployment Type, unlocking the Deploy Zip and Deploy Folder versions.

Advanced DTK also unlocks the Ignore Errors checkbox in order to let the user use that SFDX parameter on deploy.


Deploy Frame has two tabs:

  • Metadata: this tab contains all parameters needed to execute a sucessfull deploy into a Salesforce environment.

  • Script Data: this tab contains an interpreter of DTK Scripting Language.


Metadata Tab


Organization: lets the user select the target Organization.

*Source (Unlock Source setting): lets the user select the Sandbox that will be referenced as SOURCE in the scripts, in case scripts are used.

Target: lets the user select the target Sandbox for the metadata deploy.

*Deployment Type (Advanced DTK setting): it allows the user to select between 3 types of deploy. Git deploy (it will deploy the complete branch, not just the changeset of the last commit), Folder deploy ( deploying all the metadata inside the folder), and Zip deploy ( deploying the componentes inside the zip and filtering by the package.xml manifest file on the inside).

  • Git

Git Url: will be filled by default with the Organization Git url provided in the creation.

Git Branch Filter: text input this field let the user filter the provided git branches when Filter button is clicked (long developments have a lot of branches at the end, so this makes it easier to find the desired branch).

Filter button: clicking this button pulls all the branches in the git and filters them so they appear in Git Branch dropdown to be selected.

Git Branch: dropdown that lets the user selects the branch to deploy. This branch will be deployed completely, not just the changeset of the last commit.

Metadata Git Folder: root /src folder where metadata is located in the git repository used. It defaults to "sfdc\src\" following the SFDX projects structure.


  • Zip

Zip File: location of the zip file to be deployed. In order to be valid, this zip file should have this structure inside:

root/src/metadataType/metadataTypeMember

It also has to contain a manifest file (package.xml) inside the src folder.


  • Folder:

Metadata Folder: location of the folder containing the metadata to be deployed. If this location contains a valid manifest file (package.xml) , it will override the creation of the manifest file using the parameters in the frame.




Pre Deploy Script File: file selected here will be read and executed before the deploy begins ( it needs to be a DTK scripting language)

Post Deploy Script File: file selected here will be read and executed after the deploy begins ( it needs to be a DTK scripting language)

Test Level: Specifies which level of deployment tests to run. Valid values are:

NoTestRunâNo tests are run. This test level applies only to deployments to development environments, such as sandbox, Developer Edition, or trial orgs. This test level is the default for development environments.

RunSpecifiedTestsâRuns only the tests that you specify in the --runtests option. Code coverage requirements differ from the default coverage requirements when using this test level. Executed tests must comprise a minimum of 75% code coverage for each class and trigger in the deployment package. This coverage is computed for each class and trigger individually and is different than the overall coverage percentage.

RunLocalTestsâAll tests in your org are run, except the ones that originate from installed managed packages. This test level is the default for production deployments that include Apex classes or triggers.

RunAllTestsInOrgâAll tests in your org are run, including tests of managed packages.

If this field is blank the default test level used is NoTestRun.

  • Metadata Template: list the templates available, this templates are created, edited and deleted in Settings â Metadata Templates.
  • Metadata Types: will show the configuration of the metadata template selected. User can edit this templates "on the go" for the retrieve, but this will not modify the metadata template.


Metadata Types: will show the configuration of the metadata template selected. User can edit this templates "on the go" for the retrieve, but this will not modify the metadata template.

Selected Metadata Types: this window will show already selected metadata types in order to detect if you have miscliked. Useful when deploying few metadata types.

Check Only: if this checkbox is clicked then the deploy process will behave as it should, but when it finishes it will not apply any changes into the target environment. (Pre and Post script will execute completely regardless of this setting).

Ignore Warnings: Deployment will ignore warnings if checked. Otherwise the deployment will fail if any warning occurs.

*Ignore Errors (Advanced DTK): Deployment will ignored errors warnings if checked. Otherwise the deployment will fail if any errors occurs.

Confirm Deploy: if unchecked the deploy will not be executed, the git branch will be cloned and metadata will be prepared to be deployed. The pre and post scripts will execute completely.

Generate Manifest: if checked, DTK will generate a manifest file based on the metadata types selected.It will also use the API selected on the settings.

Clear Logs: while this checkbox is checked the Log window will be cleared when Deploy button is clicked.

Wait time: this box acceps number input. It will let the user select a waiting time in minutes for the deploy Job. By default it contains "-1" this mean infinite waiting.

This Deploy frame will wait until deploy is finished, refreshing automatically the log, and unziping the metadata at the end. If other number is selected, then DTK will only wait the number provided in minutes, while it is waiting it will refresh automatically the log and execute the post script. If the wait time expires then the refresh log will be unlocked, allowing the user to change the environment active in the header and deploying into another org ( we recommend to do this in another Deploy frame window separately).


Deploy button: Â this button will launch the Deploy. First it will execute all lines in pre script file, if it exists, then the deploy, and last the post deploy script if it exists.

Log: will show relevant information regarding the deploy process.

Stop button: will stop the deploy process as soon as possible for SFDX.

Refresh Log button: this will launch a SFDX report command in order to receive some feedback about the current deploy status.



Script Data Tab


Organization: lets the user select the target Organization.

*Source (Unlock Source setting): lets the user select the Sandbox that will be referenced as SOURCE in the scripts, in case scripts are used.

Target: lets the user select the target Sandbox for the metadata deploy.

File: it lets the user load files into the workspace selected so they can be used by the script.

Script File: lets the user load a script into the Script Text.

Clear Logs: while this checkbox is checked the Log window will be cleared when Deploy button is clicked.

Script Text: this box will contain the script to execute. It has to be in DTK Scripting Language.

Run Script button: Â this button will launch the script described in the Script Text.

Log: will show relevant information regarding the run script process.

Stop button: will stop the runscript process as soon as possible for SFDX.
